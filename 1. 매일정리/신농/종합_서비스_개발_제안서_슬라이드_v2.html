1. 프로젝트 개요 (Overview)

02
프로젝트 요약
로컬 고성능 GPU(RTX 5080)를 활용하여 RGB 이미지 기반의 사과 품질(당도/산도/경도) 예측 AI 서비스를 구축합니다. 보안을 위해 Cloudflare Tunnel을 적용하고, 1개월 내에 웹/모바일 서비스 MVP를 완성하여 운영하는 것을 목표로 합니다.

핵심 목표 (Goals)

  • End-to-End 서비스: 촬영 → 업로드 → 추론 → 결과 확인 → 저장의 전 과정 자동화
  • 데이터 자산화: 원본 이미지 및 예측값을 체계적으로 저장하여 향후 모델 고도화 기반 마련
  • 비용 효율화: 고가의 클라우드 GPU 대신 보유한 RTX 5080 자원 활용

주요 기술 사양 (Specs)

구분 상세 내용
GPU NVIDIA RTX 5080 (Local Server)
Network Cloudflare Tunnel (Zero Trust)
Frontend Next.js (Netlify 배포)
Input RGB Only (1차 MVP)

2. 외부 접속 방안 분석 (1/3)

03

로컬 서버를 외부 서비스로 제공하기 위해 고려된 3가지 방안별 분석 내용입니다.

방안 1: 공유기 포트포워딩 (Port Forwarding)

[분석 결과: 부적합] 보안 리스크가 매우 높고 ISP 제약으로 인한 운영 불안정성이 존재함.
  • 보안 위험 (High Risk): 로컬 네트워크의 포트를 인터넷에 직접 개방(Inbound Open)하므로, 무차별 대입 공격 및 스캐닝 봇의 표적이 됩니다.
  • 기술적 제약: 통신사(ISP)가 제공하는 네트워크가 CGNAT(사설 IP 대역)일 경우 포트포워딩 자체가 불가능할 수 있습니다.
  • HTTPS 미지원: 로컬 HTTP 서버를 노출할 경우, HTTPS 기반 프론트엔드와 혼합 콘텐츠(Mixed Content) 오류를 일으킵니다.

2. 외부 접속 방안 분석 (2/3)

04

방안 2: 전체 클라우드 이전 (Full Cloud)

  • 장점: 높은 가용성과 전문적인 보안 인프라 활용 가능.
  • 단점: RTX 5080급 GPU 인스턴스(V100 등) 임대 시 막대한 비용 발생 (월 수백만원 예상). 데모나 간헐적 사용 시 비용 효율이 매우 낮음.

방안 3: 터널링 서비스 (Cloudflare Tunnel) - [최종 채택]

[분석 결과: 최적 솔루션] 보안성, 비용 효율성, 안전성을 모두 충족하는 현대적인 아키텍처.
  • 작동 원리: 로컬 서버가 외부(Cloudflare)로 나가는(Outbound) 연결을 생성하여 트래픽을 수신합니다. 인바운드 포트를 열 필요가 없습니다.
  • 장점: ISP환경(CGNAT)과 무관하게 고정 도메인(api.yourdomain.com) 할당 가능, HTTPS 자동 적용, Zero Trust 보안 정책 적용 가능.

3. 시스템 상세 아키텍처

06
User [Web/Mobile] │ (HTTPS) ▼ Frontend [Next.js on Netlify] │ (API Request) ▼ Cloudflare Edge [api.yourdomain.com] │ (Secure Tunnel) ▼ Local Server [Cloudflared Daemon] │ (Forward) ▼ Backend [FastAPI + RTX 5080 Inference] │ ├── Local Disk [Raw Images] └── SQLite DB [Meta Data]

인증 및 보안 (RBAC)

  • SSO: NextAuth 활용 Google 로그인 연동
  • RBAC (3단계 권한):
    • OPERATOR: 시스템 전체 관리, 권한 부여
    • REVIEWER: 데이터 검수, 라벨 수정
    • USER: 이미지 업로드 및 본인 결과 조회

데이터 흐름 제어

  • Upload Limit: Cloudflare 프록시 안정성을 위해 100MB 제한
  • Pre-processing: FastAPI 내부에서 이미지 전처리 수행
  • Inference: RTX 5080 GPU 가속을 통한 실시간 추론

4. 데이터 구축 및 학습 전략 (1/2)

07

데이터 확보 계획

  • 현황: 약 1,600장 보유
  • 목표: 총 10,000장 확보 (2주간 집중 촬영)
  • 촬영 일정: 일일 600장 목표, 다양한 조명/배경/각도 환경 구성

데이터 분할 정책 (Data Split Strategy) - 중요

'사과 ID' 기반 분할 원칙
동일한 사과를 다각도에서 촬영한 이미지들이 학습(Train) 셋과 평가(Test) 셋에 섞여 들어가는 데이터 누수(Data Leakage)를 방지하기 위해, 반드시 이미지 단위가 아닌 '사과 개체 ID'를 기준으로 데이터셋을 분리합니다.

※ 이를 통해 모델이 특정 사과의 외형을 외우는 것이 아니라, 품질 특성을 실제로 학습했는지 객관적으로 검증할 수 있습니다.

4. 데이터 구축 및 학습 전략 (2/2)

08

모델링 상세 전략

1. 입력 데이터 (Input)

  • RGB Only: 1차 MVP는 일반 사용자 접근성을 위해 스마트폰 촬영(RGB) 이미지 기반 모델에 집중합니다.
  • NIR 데이터: 촬영 시 확보는 하되, 이번 추론 서비스에는 미반영 (데이터 자산 축적 후 2차 고도화 활용).

2. 출력 데이터 (Output)

  • 회귀(Regression): 당도(Brix), 산도(%), 경도(kg/cm²) 3가지 수치를 직접 예측합니다.
  • 등급 변환: 예측된 수치를 설정된 임계값(Threshold)과 비교하여 등급(A/B/C)을 화면에 표시합니다.

해상도 및 아키텍처

  • Resolution: 224x224 ~ 512x512 (RTX 5080 메모리를 활용하여 최적 해상도 탐색)
  • Backbone: EfficientNet 또는 ResNet 계열을 사용하여 속도와 정확도 균형 확보

5. 데이터베이스 및 저장소 설계

09

메타데이터 관리는 초기 비용 절감을 위해 SQLite를 사용하며, 향후 확장성을 고려해 설계합니다.

주요 테이블 스키마

테이블 주요 컬럼 및 용도
users id, google_sub, email, role(OPERATOR/REVIEWER/USER)
사용자 인증 정보 및 권한 관리
requests id(UUID), user_id, file_path(RGB/NIR), status
prediction_result (brix, acidity, firmness)
촬영 요청 기본 정보 및 AI 예측 결과 저장
label_audit id, request_id, old_value, new_value, changed_by
검수자가 라벨(실측값)을 수정할 경우 이력을 남겨 데이터 신뢰성 확보
파일 저장소 구조 (Local Disk)
/storage/raw/{YYYY-MM-DD}/{request_id}/image.jpg
DB에는 파일의 절대 경로만 저장하고, 실제 바이너리 파일은 로컬 디스크에 체계적으로 분류하여 저장합니다.
