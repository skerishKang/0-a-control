# 사과 품질 예측 AI 서비스 구축 통합 종합보고서

# 1. Executive Summary
## 1.1 최종 목표
- **목표**: 사용자가 웹/모바일에서 사과 이미지를 업로드(또는 촬영)하면
  - 로컬 서버에서 AI가 **품질 지표(회귀값)** 를 예측
  - 결과를 화면에 표시
  - 원본 이미지/예측값/메타데이터를 **영구적으로 저장**
  - 관리자가(모바일 포함) 데이터 조회/다운로드/라벨 입력·수정 및 이력 관리까지 수행

## 1.2 핵심 아키텍처
- **Netlify(Next.js) 프론트** + **Cloudflare Tunnel(고정 도메인)** + **로컬 FastAPI(RTX 5080)** + **로컬 파일스토리지 + SQLite(메타DB)**

## 1.3 가장 중요한 의사결정
- **외부 접속 방식**: 포트포워딩/공인IP 직접 노출이 아니라 **Cloudflare Tunnel** 채택
- **MVP 모델 입력**: “추론/테스트 시 NIR이 항상 없을 수 있음” → **RGB-only MVP**를 1차로 고정
- **NIR 전략**: NIR은 **데이터 자산으로 저장/축적**(2차 고도화에서 듀얼 모델 또는 고급 전략)
- **외부 모델/서비스 미사용**: 외부 AI API 호출 없이 **로컬에서 코드로 학습/추론**(모델 가중치도 자체 학습)
- **등급(A/B/C) 정책**: “등급은 바뀔 수 있다” → **연속값(pred)** 을 원본으로 저장하고 등급은 **정책 레이어(버전 포함)** 로 분리
- **권한 모델**: Google 로그인 기반 + **3단계 RBAC**
  - OPERATOR(운영자) / REVIEWER(검수자) / USER(일반)

## 1.4 일정/예산
- **일정**: 4주(1개월) MVP
  - 1~3주차: 데이터 수집/정제 + 모델 학습/개선(2~3주 집중) + 백엔드/웹 배포 병행 + 중간 테스트/문서화 상시
  - 4주차: 통합 테스트/시연 리허설/문서 마감(테스트 전용 주간)
- **데이터 확장 목표(권장)**: 2주 동안 총 **10,000장** 목표(기존 약 1,600장(회의자료 기준 RGB 1,660장) + 추가 8,400장, 일 600장 수준)
- **예산(기본 1천만원)**: 인건비 800 + 인프라/운영 제공비 200 (또는 집행 방식에 따라 “월과금형 부대비”로 200 구성 가능)
- **클라우드 대비 비용 근거(예시)**: 300시간 사용 가정 시 (환율 1,470원)
  - GCP 기준 VM+T4 약 27만원, VM+V100 약 121만원(단, 스토리지/트래픽 제외)
  - 로컬 RTX 5080 활용으로 비용/리드타임을 절감하고 정액 제공비 200으로 산정

# 2. 배경 및 현재 상황 정리
## 2.1 1개월 MVP에서 가장 중요한 4가지
- 본 단계에서 핵심은 다음 4가지입니다.
  - 1) 1개월 안에 끝낼 수 있는가
  - 2) 학습은 어떻게 시킬 것인가
  - 3) 백엔드는 어떻게 만들 것인가
  - 4) 최종 추론 결과를 어떻게 시연할 것인가
- 본 보고서는 위 4가지를 중심으로 범위/일정/학습/서빙/시연을 정리합니다.

## 2.2 1개월 안에 끝낼 수 있는가
- 결론: 범위를 “RGB-only MVP”로 고정하고, 4주 WBS대로 진행하면 1개월 완주가 가능합니다.
- 성공 조건:
  - 1주차에 학습/서빙 뼈대와 데이터 규격을 고정하고 E2E를 조기에 연결
  - 2~3주차는 학습 반복 개선에 집중(중간 테스트 포함)
  - 4주차는 통합 테스트/시연 리허설/문서 마감에 전용
- 주요 리스크 및 대응은 `# 15. 리스크 및 대응`에서 별도로 정리합니다.

## 2.3 학습은 어떻게 시킬 것인가
- 원칙:
  - 외부 모델/외부 AI API를 사용하지 않음
  - 모델 가중치는 로컬에서 자체 데이터로 학습
- 방식:
  - 사과 ID 단위로 Train/Val/Test를 분리하고, 베이스라인 → 반복 개선으로 성능을 끌어올림
  - 매 실험마다 MAE/RMSE 리포트를 생성하고, 중간 테스트로 실제 촬영 시나리오를 확인
- 자세한 내용은 `# 7. 모델링/평가 전략`의 `7.5~7.10`에 정리합니다.

## 2.4 백엔드는 어떻게 만들 것인가
- 로컬 FastAPI로 `/predict`를 제공하고, 업로드 → 저장 → 전처리 → 추론 → DB 기록 → 응답을 하나의 파이프라인으로 완성합니다.
- SQLite(메타DB)와 로컬 파일 저장소(원본)를 함께 사용하며, RBAC와 관리자 기능을 포함합니다.
- 자세한 내용은 `# 8. 백엔드 설계`, `# 10. 인증/권한 설계`, `# 11. 저장소/DB 설계`에 정리합니다.

## 2.5 최종 추론 결과를 어떻게 시연할 것인가
- 시연 흐름:
  - 사용자 로그인 → 이미지 업로드 → 예측 결과 표시 → 히스토리 조회
  - 관리자 화면에서 요청 조회/원본 다운로드/라벨 입력·수정/감사로그 확인
- 외부망 시연을 위해 `app.<도메인>`(프론트)와 `api.<도메인>`(로컬 FastAPI)을 도메인 기반으로 연결합니다.

## 2.6 로컬 GPU와 서버를 사용하는 이유
- **핵심 이유**: 이미 보유한 고성능 GPU로 학습/추론을 수행하여
  - **GPU/서버 임차비 누적**을 회피하고
  - **시연/개발 시점에만 켜는 유연성**을 확보

## 2.7 Cloudflare Tunnel을 사용하는 이유
- 포트포워딩(공인 IP 직접 노출)은 다음 현실 문제가 큽니다.
  - **기술적 제약**: ISP CGNAT/대칭 NAT 환경에서 인바운드 포트포워딩이 불안정/불가할 수 있음
  - **브라우저/HTTPS 문제**: Netlify(HTTPS) → 로컬 HTTP 호출 시 Mixed Content/CORS 이슈 발생 가능
  - **운영 리스크**: 상시 노출 시 통제 불가능한 스캐닝/트래픽 유입 및 장애 유발 가능
- 따라서 인바운드 포트 개방 없이, 로컬에서 outbound로 연결을 만드는 Cloudflare Tunnel을 채택합니다.

# 3. 범위 정의
## 3.1 1개월 MVP
### 사용자 기능
- 로그인: Google 로그인(NextAuth)
- 입력: 이미지 업로드(모바일/PC), 가능하면 모바일 카메라 촬영 UX 포함
- 출력: 예측 결과(회귀값) 표시
- 히스토리: 내 업로드 기록/결과 조회
- 원본 관리: 내 원본 이미지 조회/다운로드(정책에 따라)

### 관리자 기능
- 모바일에서도 동일하게 사용 가능하도록 반응형으로 구현
- 사용자/권한: 사용자 목록, 역할 변경(OPERATOR만 가능)
- 요청/데이터 관리: 요청 목록/검색/필터/상세
- 원본 다운로드: RGB 원본 다운로드, (존재 시) NIR 원본 다운로드
- 라벨 입력/수정: 실측값(확정된 항목) 입력 및 수정
- 감사 로그(Audit): 누가 언제 어떤 값을 바꿨는지 기록/조회

### 백엔드/저장
- 추론 API: `/predict` (업로드 → 저장 → 추론 → DB 기록 → 결과 반환)
- DB 저장: SQLite로 requests/users/labels/audit 관리
- 파일 저장: 로컬 디스크/NAS에 원본 저장(폴더 규칙 고정)
- 운영 엔드포인트: `/health`, `/version` 등

## 3.2 2차 고도화
- NIR 추론 필수 모델: “추론 시 NIR 항상 없음” 조건 때문에 MVP에선 기본 제외
- 등급 정책 확정: A/B/C 커트라인은 MVP에서 “정책 레이어(설정)”로 분리만 해두고, 확정/고도화는 2차로
- 상시(24/7) 고가용성 운영: 필요 시 2천안에서 클라우드 백엔드/DB로 단계 전환

# 4. 핵심 요구사항
## 4.1 기능 요구사항
- FR-01: 사용자 Google 로그인 및 세션 유지
- FR-02: 이미지 업로드 및 추론 요청
- FR-03: 추론 결과(회귀값) 반환 및 화면 표시
- FR-04: 원본/예측값/메타데이터 저장
- FR-05: 사용자별 히스토리 조회
- FR-06: 관리자 대시보드(모바일 포함)
- FR-07: 역할 기반 접근제어(RBAC)
- FR-08: 라벨 입력/수정 + 감사로그

## 4.2 비기능 요구사항
- NFR-01 외부접속 안정성: LTE 등 외부망에서 도메인 접속으로 데모 가능
- NFR-02 지연시간: 단건 요청에서 수 초 내 응답 수준 목표(정확치는 측정 기반 확정)
- NFR-03 데이터 신뢰성: 사과 ID 단위 split로 데이터 누출 방지
- NFR-04 확장성: SQLite → Postgres, 로컬 저장 → NAS/오브젝트 스토리지로 전환 가능해야 함
- NFR-05 운영성: 모델 버전/등급 정책 버전/로그를 남겨 재현 가능해야 함
- NFR-06 업로드 제한: Cloudflare 경로 제약을 고려해 파일당 100MB 정책(추후 측정 후 조정)

# 5. 전체 시스템 아키텍처
## 5.1 구성 요소
- 프론트엔드: Next.js (Netlify 배포)
- 인증: NextAuth + Google Provider
- 외부 접속: Cloudflare Tunnel (고정 도메인: `api.<도메인>`)
- 백엔드: FastAPI (로컬 RTX 5080 서버)
- DB: SQLite (메타데이터/권한/요청/라벨/감사로그)
- 파일 저장소: 로컬 디스크 또는 NAS (원본/가공 파일)

## 5.2 데이터 흐름
1. 사용자: `app.<도메인>` 접속 → Google 로그인
2. 업로드/촬영: 이미지 선택 또는 카메라 촬영
3. 프론트: `https://api.<도메인>/predict` 로 업로드 요청(토큰 포함)
4. Cloudflare Tunnel: 요청을 터널로 로컬 FastAPI에 전달
5. FastAPI:
    - 인증 토큰 검증
    - 업로드 검증(크기/확장자/MIME 등)
    - 원본 저장(파일시스템)
    - 전처리 → 모델 추론(RTX 5080)
    - DB 기록(SQLite)
6. 응답: 예측값 JSON 반환
7. 관리자/검수자:
    - `/admin`에서 요청 조회
    - 라벨 입력/수정 및 감사로그 확인

# 6. 데이터 전략
## 6.1 데이터 현황
- 사과 개체 수: 415개
- 촬영 각도: 4각도(0/90/180/270)
- 이미지:
    - RGB: 1,660장(=415×4)
    - NIR: 1,660장(=415×4)
    - 총 파일: 3,320
- 라벨:
    - “당도/산도/경도/강도” 4개 언급
    - 모델 출력은 3개(당도/산도/경도)로 계획

## 6.2 데이터 확장 목표
- 총 목표 장수: 10,000장
- 현재 보유: 약 1,600장(회의자료 기준 RGB 1,660장)
- 추가 목표: 8,400장
- 일일 목표(14일 기준): 약 600장/일
- 저장 증가량(대략): 8,400 × 5MB ≈ 42GB(2주 누적)

## 6.3 데이터 누출 방지 원칙
- 핵심 원칙: “사과 ID 단위”로 Train/Val/Test를 분리
- 권장 비율(기본): Train 70% / Val 15% / Test 15% (사과 개체 기준)

## 6.4 라벨 정의 불명확 대응
- 경도/강도 정의가 확정되기 전까지 유연하게 처리
- 권장안:
    - 확정 가능한 라벨(당도/산도) 우선 확정
    - 경도는 측정 방식/단위 확보 가능하면 포함
    - 강도는 정의 확인 전까지 `nullable`로 관리하고 MVP 성능 목표/출력에는 포함하지 않음

# 7. 모델링/평가 전략
## 7.1 MVP 모델 전략
- 결론: 추론/테스트에서 NIR이 항상 없으므로, MVP는 RGB-only로 완주
- NIR: 데이터 자산화 + 2차 고도화(듀얼 모델/고급 전략)

## 7.2 입력 해상도
- 권장 시작점: 224 또는 256
- 성능 부족 시: 320 → 384(필요할 때만)

## 7.3 평가 지표
- 회귀 평가지표: MAE/RMSE 중심
- 등급 평가지표: 회귀값을 커트라인으로 등급화 후 혼동행렬/리포트

## 7.4 등급 정책
- DB에는 반드시 연속 예측값(pred)을 원본으로 저장
- 등급은 설정/정책 레이어로 분리
- `grading_version`(v1/v2 등)을 함께 저장

## 7.5 모델 학습 개요
- 본 과제의 성패는 **모델 학습 품질**에 달려 있으며, 1개월 일정 중 **2~3주를 학습/개선에 집중**합니다.
- 모델 학습은 한 번에 끝나지 않습니다.
  - 데이터 품질 점검 → 학습 → 평가 → 원인 분석 → 개선을 반복해야 하며,
  - 중간 테스트 결과에 따라 데이터 수집 방향과 전처리/모델 설계가 계속 조정됩니다.
- 외부 모델이나 외부 AI API를 사용하지 않습니다.
  - 오픈소스 프레임워크는 학습 도구로만 활용하고,
  - **모델 가중치는 전부 로컬에서 자체 데이터로 학습**합니다.

## 7.6 데이터 준비
- 입력 데이터
  - RGB 이미지가 기본 입력
  - NIR 이미지는 데이터 자산으로 저장하되, MVP 추론에는 필수로 두지 않음
- 라벨
  - 당도/산도/경도는 모델 출력 대상으로 확정
  - 강도는 정의가 확정되기 전까지 보류
- 학습 분할
  - 사과 ID 단위로 Train/Val/Test를 분리하여 데이터 누출을 방지
- 품질 점검
  - 누락/중복/각도 불일치/초점 흔들림/노출 과다 등의 품질 이슈를 점검하고 리포트로 남김
  - 라벨 이상치(outlier) 후보를 자동으로 표시하여 재검수

## 7.7 모델 구조
- 목표
  - 한 장의 RGB 이미지로 당도/산도/경도 회귀값 3개를 예측
- 구현 방식
  - CNN 기반 특징 추출기 + 3개 출력 헤드(멀티 헤드 회귀)
- 학습 정책
  - 사전학습 가중치 없이 0부터 학습하여 데이터에 맞는 표현을 스스로 학습
  - 손실함수는 각 타깃의 MAE/RMSE를 조합하고, 스케일 차이는 가중치로 보정
- 과적합 방지
  - 데이터 증강, 정규화, 조기 종료, 실험 반복으로 일반화 성능을 확보

## 7.8 학습 절차
- 1단계: 베이스라인 학습
  - 소량 데이터로 학습 코드/평가 코드가 정상 동작하는지 검증
- 2단계: 정식 학습
  - Train/Val/Test 분할을 고정한 뒤 전체 데이터로 학습
- 3단계: 반복 개선
  - 2~3일 단위로 학습 결과를 분석하고 다음 실험을 설계
  - 전처리/증강/학습률/배치/에폭/모델 깊이 등을 조정
  - 실패 케이스(특정 조명/각도/크기 등)를 찾아 데이터 수집 방향을 수정
- 기록 원칙
  - 모든 실험은 설정, 데이터 버전, 결과 지표를 함께 남겨야 재현과 비교가 가능

## 7.9 중간 테스트
- 학습 중간 테스트
  - 매 실험마다 Val/Test MAE/RMSE를 비교하여 개선 여부를 판단
- 실사용 시나리오 테스트
  - 모바일 촬영 이미지로 추론을 수행하여 결과의 일관성과 체감 품질을 확인
- 속도 테스트
  - RTX 5080에서 단건 추론 시간을 측정하고, 전처리/저장/응답 병목을 개선
- 회귀 테스트
  - 모델 업데이트로 이전 성능이 급락하지 않도록 고정 샘플셋으로 비교

## 7.10 학습 산출물
- 학습 코드
  - 데이터 로더, 전처리, 학습 루프, 평가/리포트 생성 스크립트
- 재현성 자료
  - 실험 설정 파일, 시드, Train/Val/Test 분할 목록, 모델 버전 규칙
- 결과물
  - 최종 가중치 파일, 추론용 전처리 설정, 평가 리포트
