사과 품질 예측 AI 서비스 구축 통합 종합보고서

1개월 MVP · RGB-only · 로컬 학습/추론 · 관리자/감사로그 · 도메인 기반 외부 시연

1 Executive Summary

기간
4주
학습 2~3주 집중 + 4주차 통합 테스트
MVP 입력
RGB-only
추론/테스트에 NIR이 항상 없을 수 있음
서빙 위치
로컬 RTX 5080
FastAPI 기반 / 서버 시작 시 모델 로드

최종 목표

  • 사용자가 웹/모바일에서 사과 이미지를 업로드(또는 촬영)하면 로컬 서버에서 AI가 품질 지표(회귀값)를 예측
  • 결과를 화면에 표시
  • 원본 이미지/예측값/메타데이터를 영구적으로 저장
  • 관리자가(모바일 포함) 데이터 조회/다운로드/라벨 입력·수정 및 이력 관리까지 수행

핵심 아키텍처

Netlify(Next.js) 프론트 Cloudflare Tunnel(고정 도메인) 로컬 FastAPI(RTX 5080) 로컬 파일스토리지 + SQLite

가장 중요한 의사결정

외부 접속 방식

포트포워딩/공인IP 직접 노출이 아니라 Cloudflare Tunnel 채택

모델 입력

추론/테스트에서 NIR이 항상 없을 수 있으므로 RGB-only MVP로 완주

외부 모델/서비스 미사용

외부 AI API 호출 없이 로컬에서 코드로 학습/추론(모델 가중치도 자체 학습)

등급 정책 분리

연속값(pred)을 원본으로 저장하고 등급은 정책 레이어(버전 포함)로 분리

일정/예산

  • 일정: 4주(1개월) MVP
  • 1~3주차: 데이터 수집/정제 + 모델 학습/개선(2~3주 집중) + 백엔드/웹 배포 병행 + 중간 테스트/문서화 상시
  • 4주차: 통합 테스트/시연 리허설/문서 마감(테스트 전용 주간)
  • 데이터 확장 목표(권장): 2주 동안 총 10,000장 목표
  • 예산(기본 1천만원): 인건비 800 + 인프라/운영 제공비 200

2 배경 및 현재 상황 정리

핵심: 본 단계에서는 “1개월 완주 가능성 / 학습 방식 / 백엔드 구현 / 최종 시연”이 가장 중요하며, 본 보고서는 그 4가지를 중심으로 정리합니다.

1개월 MVP에서 가장 중요한 4가지

  • 1) 1개월 안에 끝낼 수 있는가
  • 2) 학습은 어떻게 시킬 것인가
  • 3) 백엔드는 어떻게 만들 것인가
  • 4) 최종 추론 결과를 어떻게 시연할 것인가

1개월 안에 끝낼 수 있는가

  • 결론: 범위를 “RGB-only MVP”로 고정하고, 4주 WBS대로 진행하면 1개월 완주가 가능합니다.
  • 성공 조건: 1주차 E2E 조기 연결 / 2~3주차 학습 반복 / 4주차 통합 테스트 전용

학습은 어떻게 시킬 것인가

  • 원칙: 외부 모델/외부 AI API를 사용하지 않음, 모델 가중치는 로컬에서 자체 데이터로 학습
  • 방식: 사과 ID 단위 Train/Val/Test 분리, 베이스라인 → 반복 개선, 매 실험 MAE/RMSE 리포트

백엔드는 어떻게 만들 것인가

  • 로컬 FastAPI로 /predict 제공
  • 업로드 → 저장 → 전처리 → 추론 → DB 기록 → 응답을 하나의 파이프라인으로 완성
  • SQLite(메타DB) + 로컬 파일 저장소 + RBAC + 관리자 기능

최종 추론 결과를 어떻게 시연할 것인가

  • 사용자: 로그인 → 업로드 → 예측 결과 → 히스토리
  • 관리자: 요청 조회/다운로드/라벨 수정/감사로그
  • 외부망 시연: app.<도메인>(프론트)와 api.<도메인>(로컬 FastAPI)을 도메인 기반 연결

로컬 GPU와 서버를 사용하는 이유

  • 이미 보유한 고성능 GPU로 학습/추론을 수행하여 GPU/서버 임차비 누적을 회피
  • 시연/개발 시점에만 켜는 유연성 확보

Cloudflare Tunnel을 사용하는 이유

  • ISP CGNAT/대칭 NAT 환경에서 포트포워딩 불안정/불가 가능
  • Netlify(HTTPS) → 로컬 HTTP 호출 시 Mixed Content/CORS 이슈 발생 가능
  • 상시 노출 시 통제 불가능한 트래픽/장애 리스크 존재
  • 따라서 인바운드 포트 개방 없이 outbound 연결 기반의 Tunnel 채택

3 범위 정의

1개월 MVP

사용자 기능

  • 로그인: Google 로그인(NextAuth)
  • 입력: 이미지 업로드(모바일/PC), 가능하면 모바일 카메라 촬영 UX 포함
  • 출력: 예측 결과(회귀값) 표시
  • 히스토리: 내 업로드 기록/결과 조회
  • 원본 관리: 내 원본 이미지 조회/다운로드(정책에 따라)

관리자 기능

  • 모바일에서도 동일하게 사용 가능하도록 반응형으로 구현
  • 사용자/권한: 사용자 목록, 역할 변경(OPERATOR만 가능)
  • 요청/데이터 관리: 요청 목록/검색/필터/상세
  • 원본 다운로드: RGB 원본 다운로드, (존재 시) NIR 원본 다운로드
  • 라벨 입력/수정: 실측값(확정된 항목) 입력 및 수정
  • 감사 로그: 누가 언제 어떤 값을 바꿨는지 기록/조회

백엔드/저장

  • 추론 API: /predict (업로드 → 저장 → 추론 → DB 기록 → 결과 반환)
  • DB 저장: SQLite로 requests/users/labels/audit 관리
  • 파일 저장: 로컬 디스크/NAS에 원본 저장(폴더 규칙 고정)
  • 운영 엔드포인트: /health, /version 등

2차 고도화

  • NIR 추론 필수 모델은 MVP에선 기본 제외(추론 시 NIR 항상 없음 가능성)
  • 등급 정책은 정책 레이어로 분리만 하고 확정/고도화는 2차로
  • 상시 운영은 필요 시 Postgres/오브젝트 스토리지로 단계 전환

FR-06

관리자 대시보드(모바일 포함)

FR-07

역할 기반 접근제어(RBAC)

FR-08

라벨 입력/수정 + 감사로그

비기능 요구사항

  • NFR-01 외부접속 안정성: LTE 등 외부망에서 도메인 접속으로 데모 가능
  • NFR-02 지연시간: 단건 요청에서 수 초 내 응답 수준 목표(정확치는 측정 기반 확정)
  • NFR-03 데이터 신뢰성: 사과 ID 단위 split로 데이터 누출 방지
  • NFR-04 확장성: SQLite → Postgres, 로컬 저장 → NAS/오브젝트 스토리지로 전환 가능해야 함
  • NFR-05 운영성: 모델 버전/등급 정책 버전/로그를 남겨 재현 가능해야 함
  • NFR-06 업로드 제한: Cloudflare 경로 제약을 고려해 파일당 100MB 정책(추후 측정 후 조정)

5 전체 시스템 아키텍처

구성 요소

  • 프론트엔드: Next.js (Netlify 배포)
  • 인증: NextAuth + Google Provider
  • 외부 접속: Cloudflare Tunnel (고정 도메인: api.<도메인>)
  • 백엔드: FastAPI (로컬 RTX 5080 서버)
  • DB: SQLite (메타데이터/권한/요청/라벨/감사로그)
  • 파일 저장소: 로컬 디스크 또는 NAS (원본/가공 파일)

데이터 흐름

  1. 사용자: app.<도메인> 접속 → Google 로그인
  2. 업로드/촬영: 이미지 선택 또는 카메라 촬영
  3. 프론트: https://api.<도메인>/predict 로 업로드 요청(토큰 포함)
  4. Cloudflare Tunnel: 요청을 터널로 로컬 FastAPI에 전달
  5. FastAPI: 인증 토큰 검증 → 업로드 검증 → 원본 저장 → 전처리/추론 → DB 기록
  6. 응답: 예측값 JSON 반환
  7. 관리자/검수자: 요청 조회 → 라벨 입력/수정 → 감사로그 확인

7 모델링/평가 전략

MVP 모델 전략

  • 결론: 추론/테스트에서 NIR이 항상 없으므로, MVP는 RGB-only로 완주
  • NIR: 데이터 자산화 + 2차 고도화(듀얼 모델/고급 전략)

입력 해상도

  • 권장 시작점: 224 또는 256
  • 성능 부족 시: 320 → 384(필요할 때만)

평가 지표

  • 회귀 평가지표: MAE/RMSE 중심
  • 등급 평가지표: 회귀값을 커트라인으로 등급화 후 혼동행렬/리포트

등급 정책

  • DB에는 반드시 연속 예측값(pred)을 원본으로 저장
  • 등급은 설정/정책 레이어로 분리
  • grading_version(v1/v2 등)을 함께 저장

모델 학습 개요

  • 1개월 일정 중 2~3주를 학습/개선에 집중
  • 데이터 품질 점검 → 학습 → 평가 → 원인 분석 → 개선 반복
  • 외부 모델이나 외부 AI API를 사용하지 않음

데이터 준비 / 모델 구조 / 학습 절차 / 중간 테스트 / 산출물

본문은 원문 보고서 구조를 유지하며, 세부 항목은 동일 내용을 반영합니다.

  • 데이터 준비: RGB 기본 입력, NIR 자산 저장, 사과 ID split, 품질 점검
  • 모델 구조: CNN 특징 추출기 + 3개 출력 헤드(멀티 헤드 회귀)
  • 학습 절차: 베이스라인 → 정식 학습 → 반복 개선(2~3일 단위 실험)
  • 중간 테스트: Val/Test 지표, 실사용 촬영 시나리오, 속도/회귀 테스트
  • 학습 산출물: 코드/설정/분할 목록/리포트/가중치

13 4주 WBS

Week 1 — 착수/데이터 규격/학습·서빙 뼈대

  • 데이터 수집 규칙, 파일/라벨 규격 확정
  • 사과 ID split 리스트 고정, 품질 점검 리포트 자동화
  • 학습 코드 베이스라인 구현, 소량 데이터로 1차 학습
  • 백엔드 뼈대(FastAPI, 업로드, 저장, DB 스키마, 권한) 구성
  • 프론트 뼈대(로그인, 업로드 화면) 구성, 웹 배포 준비

Week 2 — 데이터 확장/모델 학습 1차/추론 연동

  • 데이터 수집·정제 본격화(일 600장 목표)
  • 모델 학습 1차 및 추론 모듈 구현
