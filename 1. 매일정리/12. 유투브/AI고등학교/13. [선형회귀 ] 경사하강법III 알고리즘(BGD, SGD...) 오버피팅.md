# Gradient Descent 구현과 Overfitting 강의 요약

## 그래디언트 계산의 실제

### 손실함수의 그래디언트
**손실함수**: $L(w)$는 w에 대한 **스칼라 함수** (모든 데이터의 평균값)

**개별 데이터에 대한 그래디언트**:
$$\frac{\partial L_i}{\partial w} = (w^T x_i - y_i) \cdot x_i$$

**구성요소 분석**:
- $w^T x_i$: **예측값** (스칼라)
- $y_i$: **실제값** (스칼라)
- $(w^T x_i - y_i)$: **오차** (스칼라)
- $x_i$: **입력벡터** (열벡터)
- **결과**: 스칼라 × 벡터 = **벡터**

### 벡터 표기법의 실용적 처리
- **이론**: 스칼라를 벡터로 미분하면 **행벡터**
- **실무**: 머신러닝에서는 편의상 **열벡터**로 취급
- **이유**: 코드 구현과 수학적 일관성

> 🔧 **주석**: 대부분의 ML 라이브러리에서 그래디언트를 열벡터로 처리하여 일관성 유지

## Gradient Descent 알고리즘

### 기본 업데이트 공식
$$w_{\text{new}} = w_{\text{old}} - \gamma \nabla L(w_{\text{old}})$$

**핵심 요소**:
- **마이너스 부호**: 손실 **감소** 방향으로 이동
- **$\gamma$ (학습률)**: 이동 **보폭** 조절
- **$\nabla L(w)$**: **가장 가파른 증가** 방향

### 기하학적 이해
**2차원 예시**:
- **그래디언트 > 0**: 오른쪽이 증가 방향 → 왼쪽으로 이동
- **그래디언트 < 0**: 왼쪽이 증가 방향 → 오른쪽으로 이동
- **마이너스 부호**: 항상 **감소 방향**으로 유도

### 계산 복잡도 비교
**Direct Solution**:
- **역행렬 연산**: $O(d^3)$
- **메모리**: $(X^TX)^{-1}$ 저장 필요

**Gradient Descent**:
- **각 업데이트**: $O(nd)$ (n: 데이터 수, d: 특성 수)
- **역행렬 없음**: 단순한 덧셈과 곱셈만
- **결과**: 고차원에서 **훨씬 효율적**

## 학습률 (Learning Rate)

### 학습률의 중요성
**너무 작은 경우**: 
- **문제**: 매우 느린 수렴
- **예시**: $\gamma = 0.0001$

**너무 큰 경우**:
- **문제**: 발산 또는 최솟값 **오버슈트**
- **결과**: 수렴 실패

### 학습률 선택 전략

#### 1. Grid Search
**방법**: 여러 값을 체계적으로 시도
```
γ ∈ {0.001, 0.01, 0.1, 1.0}
```

#### 2. 적응적 학습률
**시간 기반 감소**:
$$\gamma_t = \frac{\gamma_0}{t}$$
- **초기**: 빠른 수렴
- **후기**: 안정적 수렴

#### 3. 실험적 조정
- **발산 감지**: 손실이 증가하면 학습률 감소
- **너무 느림**: 진전이 없으면 학습률 증가

### 수렴 조건
**이론적**: $\nabla L(w) = 0$
**실용적**: $|\nabla L(w)| < \epsilon$ (예: $\epsilon = 0.001$)

**이유**:
- **부동소수점 오차**: 정확히 0 달성 어려움
- **계산 효율성**: 무한 반복 방지
- **실용적 충분성**: 작은 그래디언트면 충분히 좋은 해

## Gradient Descent의 변형들

### 1. Batch Gradient Descent (BGD)

#### 알고리즘
```
전체 데이터로 그래디언트 계산:
∇L(w) = (1/N) Σ(wᵀxᵢ - yᵢ)xᵢ

업데이트:
w ← w - γ∇L(w)
```

#### 특징
**장점**:
- **안정적 수렴**: 부드러운 경로
- **정확한 그래디언트**: 전체 데이터의 평균

**단점**:
- **느린 계산**: 한 번 업데이트에 모든 데이터 필요
- **메모리 사용량**: 대용량 데이터에서 문제

### 2. Stochastic Gradient Descent (SGD)

#### 알고리즘
```
각 데이터 xᵢ에 대해:
∇Lᵢ(w) = (wᵀxᵢ - yᵢ)xᵢ

업데이트:
w ← w - γ∇Lᵢ(w)
```

#### 특징
**장점**:
- **빠른 반응**: 좋은 데이터를 만나면 즉시 수렴
- **메모리 효율**: 한 번에 하나씩만 처리
- **탈출 능력**: Local minima, Saddle point 탈출 가능

**단점**:
- **불안정한 경로**: 지그재그 움직임
- **최솟값에서 진동**: 정확한 수렴 어려움

#### 중요한 요구사항: 데이터 셔플링
**나쁜 예**: 비슷한 데이터만 연속 학습
- **편향된 학습**: 특정 패턴에만 치우침
- **조기 종료**: 잘못된 수렴

**좋은 예**: 다양한 데이터 골고루 섞음
- **대표성**: 전체 데이터 패턴 학습
- **올바른 수렴**: 일반화 성능 향상

#### 수학적 보장
**Convex 함수**에서 적절한 학습률 사용 시 **수렴 보장** (수학적으로 증명됨)

### 3. Mini-batch Gradient Descent

#### 알고리즘
```
배치 크기 B로 데이터 분할:
각 미니배치 Bₖ에 대해:
∇L(w) = (1/B) Σ(wᵀxᵢ - yᵢ)xᵢ (i ∈ Bₖ)

업데이트:
w ← w - γ∇L(w)
```

#### 특징
**절충안의 장점**:
- **BGD보다 빠름**: 전체 데이터 기다릴 필요 없음
- **SGD보다 안정적**: 배치 평균으로 변동성 감소
- **하드웨어 최적화**: GPU 병렬 처리에 이상적

**하이퍼파라미터**: 배치 크기 (보통 32, 64, 128, 256)

---

## 수렴 패턴 비교

### 시각적 차이
**BGD**: 
- **부드러운 곡선**: 직선적이고 안정적
- **예측 가능**: 항상 같은 방향

**SGD**: 
- **지그재그 패턴**: 불규칙한 움직임
- **빠른 초기 수렴**: 운이 좋으면 BGD보다 빠름
- **최솟값 근처 진동**: 정확한 수렴 어려움

**Mini-batch**: 
- **중간적 성격**: SGD보다 부드럽고 BGD보다 빠름

### SGD의 진동 해결법
**학습률 감소**:
$$\gamma_t = \frac{\gamma_0}{t}$$
- **후반부**: 학습률이 0에 가까워져 **거의 움직이지 않음**
- **결과**: 최솟값 근처에서 **안정적 수렴**

## 딥러닝에서의 복잡성

### 고차원 최적화의 어려움
**매개변수 규모**: 수백만~수조 개
**손실함수**: 매우 복잡하고 험준한 지형
**차원**: 시각화 불가능한 초고차원

### 특별한 문제들

#### Saddle Point (안장점)
**정의**: 일부 방향은 오목, 일부 방향은 볼록
**문제**: 그래디언트가 0이지만 최솟값이 아님
**빈도**: 고차원에서 **Local minima보다 훨씬 많음**

#### Local Minima
**정의**: 주변보다 작지만 전역 최솟값이 아님
**문제**: 더 좋은 해가 존재하지만 찾지 못함

#### 차원의 저주
**고차원 특성**: 모든 방향이 동시에 아래로 볼록하기 **거의 불가능**
**결과**: Saddle point가 **압도적으로 많음**

### 발생 원인

#### 1. 모델 복잡도 과다
**선형회귀에서**: 고차 다항식 사용
```
y = w₀ + w₁x + w₂x² + ... + w₉x⁹
```
**특성**: 데이터 개수보다 많은 매개변수

#### 2. 데이터 부족
**상대적 부족**: 모델 복잡도 대비 데이터 수 부족
**절대적 부족**: 전체적으로 데이터가 적음

#### 3. 편향된 데이터
**특정 패턴**: 일부 패턴에만 치우친 데이터
**노이즈 학습**: 실제 신호보다 노이즈를 학습

### 데이터 양과 Overfitting의 관계

#### 실험적 관찰
**9차 모델 + 9개 데이터**: 심각한 Overfitting
- **문제**: 매개변수 수 = 데이터 수
- **결과**: 완벽한 훈련 성능, 끔찍한 일반화

**9차 모델 + 15개 데이터**: 어느 정도 개선
- **제약**: 더 많은 점을 지나야 함
- **결과**: 약간의 일반화 향상

**9차 모델 + 100개 데이터**: 좋은 일반화
- **강한 제약**: 100개 점을 모두 지나기 어려움
- **결과**: 진짜 패턴 학습, 노이즈 무시

#### 핵심 통찰
**"데이터가 많으면 복잡한 모델도 적절히 제약됨"**

**이유**:
- **더 많은 제약조건**: 모든 점을 지나기 어려워짐
- **패턴 vs 노이즈**: 진짜 패턴만 학습하게 됨
- **일반화 향상**: 다양한 상황 경험

## Overfitting 해결 방법

### 1. 더 단순한 모델 사용

#### 방법
**차수 감소**: 9차 → 3차 다항식
**특성 선택**: 중요한 특성만 사용
**모델 복잡도 제한**: 매개변수 수 제한

#### 주의사항
**Underfitting 위험**: 너무 단순하면 패턴을 못 잡음
**적절한 균형**: 복잡도와 일반화의 trade-off

### 2. 더 많은 데이터 확보

#### 장점
**근본적 해결**: 가장 확실한 방법
**강력한 제약**: 복잡한 모델도 일반화 가능

#### 한계
**시간과 비용**: 데이터 수집의 현실적 제약
**데이터 품질**: 양뿐만 아니라 질도 중요

### 3. 정규화 (Regularization)

#### 기본 아이디어
**복잡성에 페널티**: 모델이 너무 복잡해지는 것을 방지

**일반적 형태**:
$L_{\text{total}}(w) = L_{\text{original}}(w) + \lambda \cdot \text{Penalty}(w)$

#### L1 정규화 (Lasso Regression)
$L(w) = \text{MSE}(w) + \lambda \sum_{i=1}^{d} |w_i|$

**특징**:
- **Sparse solution**: 일부 가중치를 정확히 0으로
- **특성 선택 효과**: 중요하지 않은 특성 제거
- **해석 가능성**: 적은 수의 특성만 사용

#### L2 정규화 (Ridge Regression)
$L(w) = \text{MSE}(w) + \lambda \sum_{i=1}^{d} w_i^2$

**특징**:
- **Small weights**: 가중치를 작게 유지
- **부드러운 함수**: 급격한 변화 방지
- **수치적 안정성**: 일반적으로 더 안정적

#### 정규화 강도 조절
**$\lambda$ (람다)**: 정규화 강도 조절 하이퍼파라미터
- **$\lambda = 0$**: 정규화 없음 (원래 손실함수)
- **$\lambda$ 크면**: 강한 정규화 (단순한 모델)
- **$\lambda$ 작으면**: 약한 정규화 (복잡한 모델 허용)

#### 딥러닝의 정규화
**Dropout**: 무작위로 뉴런 비활성화
**Batch Normalization**: 배치 단위 정규화
**Early Stopping**: 검증 성능 기반 조기 종료

## 실무에서의 고려사항

### 하이퍼파라미터 튜닝

#### 주요 하이퍼파라미터들
1. **학습률** ($\gamma$): 수렴 속도와 안정성
2. **배치 크기**: 메모리와 수렴 품질의 trade-off
3. **에포크 수**: 학습 시간과 과적합의 trade-off
4. **정규화 강도** ($\lambda$): 복잡도와 성능의 trade-off

#### 튜닝 전략
**Grid Search**: 체계적이지만 계산 비용 높음
**Random Search**: 더 효율적, 실제로 자주 사용
**베이지안 최적화**: 고급 기법, 효율적 탐색

### 성능 평가

#### 데이터 분할
**훈련 세트**: 모델 학습용
**검증 세트**: 하이퍼파라미터 튜닝용
**테스트 세트**: 최종 성능 평가용

#### 교차 검증
**K-fold CV**: 데이터를 K개로 나누어 검증
**목적**: 더 안정적인 성능 추정

### 조기 종료 (Early Stopping)
**원리**: 검증 성능이 더 이상 향상되지 않으면 학습 중단
**장점**: 과적합 방지, 계산 시간 절약
**구현**: 검증 손실 모니터링, patience 매개변수

## 이론과 실무의 차이

### 이론적 보장 vs 실제 성능
**Convex 최적화**: 수학적으로 수렴 보장
**실제 문제**: 대부분 non-convex, 경험적 해결

### 하이퍼파라미터의 현실
**이론적 최적값**: 대부분 존재하지 않음
**실무적 접근**: 실험과 경험에 의존
**AutoML**: 자동화 시도, 하지만 여전히 한계

### 딥러닝에서의 미스터리
**이론적 이해 부족**: 왜 잘 되는지 완전히 이해 못함
**실제 성공**: 이론적 보장 없이도 놀라운 성능
**연구 방향**: 이론적 기반 구축 노력 지속

## 다음 학습 계획

### 실습 중심 학습
**Python 구현**: NumPy를 이용한 from scratch 구현
```python
def gradient_descent(X, y, learning_rate, epochs):
    w = np.random.randn(X.shape[1])
    for epoch in range(epochs):
        # Forward pass
        predictions = X @ w
        # Compute loss and gradient
        loss = np.mean((predictions - y)**2)
        gradient = (2/len(y)) * X.T @ (predictions - y)
        # Update weights
        w -= learning_rate * gradient
    return w
```

### 비교 실험
**알고리즘 비교**: BGD vs SGD vs Mini-batch
**학습률 실험**: 다양한 학습률의 수렴 패턴
**정규화 효과**: L1, L2 정규화의 실제 효과

### 시각화
**수렴 과정**: 손실함수의 변화 그래프
**가중치 변화**: 학습 과정에서 가중치 궤적
**결정 경계**: 2D 데이터에서의 분류 경계

### 실제 데이터셋
**Boston Housing**: 회귀 문제
**California Housing**: 대용량 데이터
**Iris**: 분류 문제로 확장

## 핵심 통찰과 교훈

### 1. 완벽한 방법은 없다
**모든 방법에 trade-off 존재**:
- BGD: 안정적이지만 느림
- SGD: 빠르지만 불안정
- Mini-batch: 절충안이지만 완벽하지 않음

### 2. 실험적 접근의 중요성
**하이퍼파라미터**: 이론적 최적값이 없어서 실험 필요
**데이터 의존성**: 데이터셋마다 최적 설정 다름
**경험의 가치**: 많은 실험을 통한 직관 축적

### 3. 단순함의 힘
**Occam's Razor**: 단순한 모델이 종종 더 좋음
**정규화**: 복잡성을 제어하는 강력한 도구
**데이터의 중요성**: 복잡한 모델보다 더 많은 데이터

### 1. "손실함수 $L(w)$는 w에 대한 스칼라 함수 (모든 데이터의 평균값)"

**쉽게 설명하면:**

우리 컴퓨터 친구가 아파트 가격을 예측해서 얼마나 틀렸는지 점수를 매긴다고 했지? 그 점수가 바로 **'손실함수(Loss Function)'** 야.

*   이 손실함수 점수는 우리 컴퓨터 친구의 **'생각'(가중치 $w$)에 따라 달라지는 점수**야. $w$가 바뀌면 점수도 바뀌는 거지.
*   그리고 이 점수는 **'모든 아파트 데이터'를 다 합쳐서 평균을 낸 점수**야. 아파트 한두 채만 보고 점수를 매기는 게 아니라, 모든 아파트를 다 본 평균 점수인 거지.

이때 손실함수 점수는 **하나의 숫자**로 나와. (예: 26만 만원 제곱!) 그래서 이걸 **'스칼라 함수'** 라고 부르는 거야. 스칼라는 '크기만 있는 하나의 숫자'를 뜻한다고 배웠지?

---

### 2. "개별 데이터에 대한 그래디언트: 
$\frac{\partial L_i}{\partial w} = (w^T x_i - y_i) \cdot x_i$"

**쉽게 설명하면:**

우리 컴퓨터 친구가 아파트 가격을 예측했는데, 이제 **왜 틀렸는지** (오차)를 알아보고, **어떻게 자기 생각($w$)을 바꿔야 하는지**를 찾아야 해.

*   **$(w^T x_i - y_i)$**: 이건 **'오차'** 야. 즉, **예측한 가격과 실제 가격이 얼마나 차이 나는지**를 나타내는 숫자지. (예: 500만원 틀렸어!)

*   **$x_i$**: 이건 **'입력 정보'** 야. 그 아파트의 '면적'이나 '방 개수' 같은 원래 정보들이지.

**이 둘을 '곱하면'** 뭐가 나올까?
*   '오차'는 하나의 숫자('스칼라')이고,
*   '입력 정보'는 여러 개의 숫자가 들어있는 '숫자 꾸러미'('벡터')잖아?

그래서 둘을 곱하면, **'오차'만큼 '입력 정보 꾸러미'가 커지거나 작아진 '새로운 숫자 꾸러미(벡터)'** 가 나와.

이 **'새로운 숫자 꾸러미(벡터)'** 가 바로 '그래디언트'인데, 이게 **'이 아파트 하나만 보고, 컴퓨터 친구의 생각($w$)을 어떤 방향으로 얼마나 바꿔야 할지'** 를 알려주는 나침반 역할을 한단다!

### 3. "실무: 머신러닝에서는 편의상 열벡터로 취급. 이유: 코드 구현과 수학적 일관성"

**쉽게 설명하면:**

전에 '벡터 미분' 배울 때,
*   **'스칼라(하나의 숫자)를 벡터(숫자 꾸러미)로 미분하면, 행벡터'** 가 나온다고 했지? (가로로 긴 숫자 꾸러미)
*   **'벡터를 스칼라로 미분하면, 열벡터'** 가 나온다고 했고. (세로로 긴 숫자 꾸러미)

이론적으로는 이렇게 가로, 세로가 중요해.

하지만 **컴퓨터 친구가 실제로 계산할 때(실무)** 는 이걸 좀 **편리하게 바꾸어서** 계산해.
*   많은 컴퓨터 프로그램들은 **'세로로 긴 숫자 꾸러미(열벡터)'** 를 더 편하게 생각하고 계산하거든.
*   그래서 원래는 가로로 나와야 하는 계산 결과도, **'우리는 그냥 세로로 보자!'** 하고 약속하고 계산하는 경우가 많아.

**왜냐하면:**
*   **'코드 구현'** 이 훨씬 편해지고,
*   모든 계산을 **'세로로 긴 꾸러미'** 로 통일하면 **'수학적으로도 더 깔끔하고 헷갈리지 않게'** 정리할 수 있거든.

그러니까 이론이랑 실제가 아주 조금 다를 수 있지만, **다 컴퓨터 친구들이 편하고 효율적으로 일하게 하려고 하는 약속**이라고 생각하면 돼!

### 4. "Direct Solution: 역행렬 연산: $O(d^3)$"

**쉽게 설명하면:**

이건 전에 '머신러닝 Direct Solution 강의'에서 설명했던 것과 똑같은 내용이야!

'Direct Solution'이라는 방법은 아파트 정보를 가지고 **정답을 딱! 한 번에 찾는 방법**인데, 이때 **'역행렬'** 이라는 아주 복잡한 계산을 해야 해.

*   '$O(d^3)$ 복잡도'라는 건, 이 '역행렬' 계산이 얼마나 힘든지 나타내는 말이야.
    *   $d$는 아파트 정보의 종류(면적, 방 개수, 층수 등)가 몇 가지나 되는지 나타내는 숫자인데, 이게 조금만 많아져도 계산할 양이 **엄청나게 불어나.**
    *   $d$가 100개면 백만 번 계산하고, $d$가 1000개면 십억 번 계산해야 하거든.

그래서 'Direct Solution'은 이론적으로는 완벽하지만, **현실에서는 너무 힘들어서 잘 사용하지 않아!**

### 5. "Gradient Descent: 각 업데이트: $O(nd)$"

**쉽게 설명하면:**

'Gradient Descent (경사 하강법)'는 컴퓨터 친구가 아파트 가격을 예측하는 **가중치($w$)를 조금씩 바꿔가면서 정답에 다가가는 방법**이라고 했지? (언덕을 조금씩 굴러 내려가는 것처럼)

*   이때, 한 번 **'조금' 움직일 때마다(업데이트)** 얼마나 많은 계산을 해야 할까?
    *   $N$은 아파트 데이터가 몇 개나 있는지 나타내는 숫자고,
    *   $D$는 아파트 정보의 종류가 몇 가지나 되는지 나타내는 숫자야.

*   $O(ND)$ 복잡도라는 건, **한 번 움직일 때마다 $N$이랑 $D$를 곱한 만큼의 계산**을 해야 한다는 뜻이야.
    *   만약 아파트 데이터가 100만 개($N=1,000,000$)이고 정보 종류가 100개($D=100$)라면, 한 번 움직일 때 $1,000,000 \times 100 = 1억$ 번 계산해야 해.

전에 'Direct Solution'은 $D^3$만큼 계산한다고 했는데, $D$가 1000개만 돼도 $1000^3=10$억 번 계산해야 했잖아?

'경사 하강법'은 **한 번 움직일 때 계산하는 양은 더 적어.** 하지만 **이걸 여러 번 반복**해야 정답에 도착할 수 있지. 그래도 'Direct Solution'보다 훨씬 더 효율적이라서, **실제 컴퓨터 친구들이 배우는 데 주로 사용되는 방법**이란다!

### 6. "수렴 조건: 이론적: $\nabla L(w) = 0$, 실용적: $|\nabla L(w)| < \epsilon$"

**쉽게 설명하면:**

컴퓨터 친구가 언덕을 굴러 내려가서 가장 낮은 바닥(최솟값)에 도착하면 **'기울기가 0'** 이 된다고 했지?
*   **이론적으로는** $\nabla L(w) = 0$이 될 때가 완벽한 최솟값이야. (기울기 나침반이 아무 방향도 가리키지 않을 때)

하지만 **실제로 컴퓨터가 계산할 때**는 좀 달라.
*   컴퓨터는 '소수점'으로 계산하는데, 아주아주 미세한 오차가 있을 수 있어. 그래서 **정확히 0이 되기는 어려워.**
*   게다가, 딱 0이 될 때까지 기다리려면 **너무너무 오래 걸릴 수도 있어.**

그래서 우리는 컴퓨터 친구한테 이렇게 말해줘:
*   "기울기가 **거의 0에 가까워지면($|\nabla L(w)| < \epsilon$) 멈춰도 돼!**"
*   여기서 $\epsilon$은 **'아주 작은 숫자'** 를 뜻해. (예: 0.001)

이건 마치 "방을 완벽하게 깨끗하게 청소해!"라고 말하는 것보다, "방이 **거의 깨끗해지면** 그만해도 돼!"라고 말하는 것과 같아. **완벽하게 0이 아니어도 충분히 좋은 답을 찾을 수 있으니까**, 시간 낭비하지 말라는 뜻이지!

### 7. "Batch Gradient Descent (BGD) vs Stochastic Gradient Descent (SGD) vs Mini-batch Gradient Descent"

**쉽게 설명하면:**

컴퓨터 친구가 언덕을 굴러 내려가서 가장 낮은 바닥을 찾을 때, **어떤 정보를 가지고 움직일지**에 따라 세 가지 방법으로 나눌 수 있어.

1.  **Batch Gradient Descent (BGD) - '전체 데이터 코치'**:
    *   컴퓨터 친구가 **모든 아파트 데이터(수천 개, 수만 개)** 를 **한꺼번에 다 보고** "아, 평균적으로 이쪽으로 움직여야겠네!" 하고 **한 번만 움직여.**
    *   **장점**: 가장 정확한 방향으로 움직여서 길이 아주 **부드럽고 안정적**이야.
    *   **단점**: 모든 데이터를 다 봐야 하니까 **한 번 움직이는 데 엄청 오래 걸려.** 마치 수천 명의 의견을 다 듣고 움직이는 것처럼.

2.  **Stochastic Gradient Descent (SGD) - '하나씩 코치'**:
    *   컴퓨터 친구가 **아파트 데이터 딱 하나만 보고** "이 아파트는 이쪽으로 움직이는 게 좋겠네!" 하고 **바로 움직여.**
    *   **장점**: 한 번 움직이는 데 **아주 빨라!** 마치 한 명의 의견만 듣고 바로 움직이는 것처럼.
    *   **단점**: 데이터 하나만 보고 움직이니까 **엉뚱한 방향으로 지그재그로 움직일 때가 많아.** 최솟값 근처에서도 계속 흔들흔들거려.

3.  **Mini-batch Gradient Descent (미니배치 GD) - '그룹 코치'**:
    *   이건 BGD와 SGD의 **중간 방법**이야. 컴퓨터 친구가 **아파트 데이터를 몇십 개, 몇백 개씩 '그룹'으로 묶어서** "이 그룹 아파트들은 이쪽으로 움직이는 게 좋겠네!" 하고 움직여.
    *   **장점**: BGD보다 훨씬 **빠르고**, SGD보다 훨씬 **안정적**이야. 컴퓨터가 계산하기에도 가장 효율적이라서, **실제로 가장 많이 사용하는 방법**이란다!

---

### 8. "SGD의 진동 해결법: 학습률 감소"

**쉽게 설명하면:**

'SGD'는 아파트 데이터 하나하나를 보고 움직이니까 **지그재그로 흔들흔들거리면서** 최솟값 근처에서 계속 왔다갔다 할 수 있다고 했지? (진동)

이걸 해결하는 방법은 **'학습률 ($\alpha$)'을 점점 줄여주는 것**이야.
*   **처음에는 학습률을 크게** 해서 빨리빨리 최솟값 근처로 가. (성큼성큼 걸어)
*   하지만 **최솟값에 가까워지면 학습률을 점점 작게** 만들어서 아주 조금씩만 움직이게 해. (살금살금 걸어)

마치 언덕을 내려갈 때, 처음에는 성큼성큼 내려가다가, 바닥에 가까워지면 발을 살금살금 디디면서 정확하게 멈추는 것과 똑같아. 이렇게 하면 **진동을 줄이고 최솟값에 더 정확하게 도착**할 수 있단다!

---

### 9. "Saddle Point (안장점) vs Local Minima vs 차원의 저주"

**쉽게 설명하면:**

우리 컴퓨터 친구가 언덕에서 가장 낮은 바닥을 찾으려고 굴러 내려가는데, 언덕 모양이 너무 복잡해서 어려움을 겪을 수 있다고 했지?

*   **'Local Minima (지역 최솟값)'**: 이건 **진짜 바닥은 아니고, 잠시 움푹 파인 작은 웅덩이**들이야. 컴퓨터 친구가 여기를 진짜 바닥으로 착각하고 멈출 수 있지.

*   **'Saddle Point (안장점)'**: 이건 말 안장처럼 생긴 곳이야. 어떤 방향으로는 **내려갈 수 있는데**, 다른 방향으로는 **올라가는 평평한 길**이 있어. 컴퓨터 친구가 "어? 여기 기울기가 0인데?" 하고 착각하고 멈출 수 있지. (이게 컴퓨터 친구들을 더 많이 헷갈리게 해!)

*   **'차원의 저주'**: 컴퓨터 친구들이 다루는 아파트 정보의 종류($D$)가 수백만 개, 수십억 개가 되는 **'고차원 공간'에서는 이런 복잡한 안장점이 훨씬 더 많아.** 진짜 바닥을 찾기가 더욱 어려워지는 거지. 마치 아주 넓고 복잡한 산에 웅덩이와 안장점이 수만 개씩 있는 것과 같아.

이런 문제들 때문에 딥러닝 친구들은 이 복잡한 언덕을 더 잘 내려가기 위해 여러 가지 똑똑한 기술들을 사용한단다.

### 10. "Overfitting (과적합) 해결 방법"

**쉽게 설명하면:**

'Overfitting (과적합)'은 컴퓨터 친구가 **공부할 때 너무 열심히 외워서, 새로운 문제를 만나면 오히려 못 푸는 것**을 말해.

*   예를 들어, 친구가 시험 공부를 할 때 문제집의 **답만 통째로 외워서** 시험을 봤다고 생각해 봐. 그럼 문제집에 있는 문제는 다 맞히겠지만, 시험에 **조금이라도 다른 문제**가 나오면 못 풀겠지?

컴퓨터 친구도 똑같아. 공부할 때 너무 훈련 데이터만 외워서, **새로운 데이터(문제)** 를 만나면 점수가 확 떨어지는 거야.

**과적합을 해결하는 방법들:**

1.  **'더 단순한 모델 사용'**: 컴퓨터 친구한테 **너무 많은 정보(복잡한 계산 방법)** 를 주지 않는 거야. "시험 문제를 외우지 말고, 가장 중요한 개념만 공부해!" 하는 것처럼 말이지.

2.  **'더 많은 데이터 확보'**: 컴퓨터 친구한테 **다양한 종류의 문제집을 많이 많이 주는 거야.** 그럼 답만 외울 수 없으니 진짜 개념을 공부하게 되겠지? 이게 가장 확실한 방법이야.

3.  **'정규화 (Regularization)'**: 이건 컴퓨터 친구가 너무 열심히 외우지 못하게 **'벌점'을 주는 방법**이야. "너무 복잡하게 외우지 마! 그러면 점수 깎는다!" 하고 벌점을 주면서, **가장 중요한 부분만 배우게** 하는 거지. (L1, L2 같은 방법이 있어)

이런 방법들로 컴퓨터 친구가 새로운 문제를 만나도 똑똑하게 잘 풀 수 있도록 도와준단다.

---

### 11. "데이터가 많으면 복잡한 모델도 적절히 제약됨"

**쉽게 설명하면:**

컴퓨터 친구가 아파트 가격을 예측하는 **복잡한 방법(모델)** 을 배웠다고 생각해 봐.

*   만약 아파트 데이터가 **아주 조금만 있다면**, 컴퓨터 친구는 그 적은 데이터에 **완벽하게 맞춰서** 자기 방법을 만들 수 있어. 마치 답이 딱 3개 있는 문제집을 주고, 그 3개의 답만 달달 외우게 하는 것과 같아. (과적합 위험!)

*   하지만 아파트 데이터가 **아주아주 많다면**, 컴퓨터 친구는 아무리 복잡한 방법이라도 **모든 데이터를 다 완벽하게 맞추기는 어려워져.** 왜냐하면 모든 데이터에는 조금씩 다른 '잡음'이나 '특별한 상황'이 있거든.

그래서 데이터가 많아지면, 컴퓨터 친구는 '잡음'이나 '특별한 상황' 같은 것들은 무시하고, **'진짜로 중요한 규칙(패턴)'만 찾아서 배우게 돼.** 마치 문제집이 엄청 많으면 답만 외울 수 없으니, 진짜 개념을 공부하게 되는 것처럼 말이야.

이렇게 **데이터가 많으면, 컴퓨터 친구가 너무 쓸데없이 외우는 것을 막아주고, 진짜 똑똑하게 배우도록 도와준다는 뜻**이야.

### 12. "Hessian의 크기는 d×d이므로, 매개변수가 많으면 계산과 저장이 불가능"

**쉽게 설명하면:**

이건 전에 '벡터 미분 강의'에서 설명했던 것과 똑같은 내용이야!

'헤시안 행렬'은 언덕의 굴곡진 정도를 알려주는 지도라고 했지?
이 지도의 크기는 우리 컴퓨터 친구가 조절하는 **'가중치(매개변수)'의 개수($D$)가 많아질수록 엄청나게 커져.**

*   만약 가중치가 $100$개라면, 지도의 크기는 $100 \times 100$ = $10000$칸이 돼.
*   만약 가중치가 $1000$개라면, 지도의 크기는 $1000 \times 1000$ = $1,000,000$ (백만)칸이 돼.
*   딥러닝 친구들은 가중치가 수백만 개, 수십억 개가 되는 경우도 흔해! 그럼 지도의 칸 수는 **상상할 수 없을 만큼 커지겠지?**

이렇게 지도의 칸 수가 너무너무 많아지면, 컴퓨터가 그 **지도를 만드는 데 드는 계산도 엄청 많아지고, 지도를 저장할 공간(메모리)도 너무 많이 필요하게 돼.**

그래서 딥러닝 친구들은 이 헤시안 지도를 만드는 게 **'불가능'** 하거나 **'너무 비싸서'** 잘 사용하지 않는단다. 대신 '그래디언트'라는 나침반만 가지고 굴러 내려가는 경우가 대부분이야.
