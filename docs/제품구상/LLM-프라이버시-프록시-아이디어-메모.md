# LLM 프라이버시 프록시 아이디어 메모

가칭: `Padiem Vault`

기업 내부의 민감 정보를 외부 LLM에 직접 보내지 않으면서도, 문맥을 유지한 상태로 LLM을 활용하게 해주는 프록시/게이트웨이 아이디어.

## 문제의식

- 직원이 ChatGPT/Claude 같은 외부 LLM을 업무에 쓰고 싶어 함
- 하지만 회사명, 인명, 금액, 프로젝트명, 계약 정보, 코드명 등 민감정보 유출 위험이 큼
- 단순 `[MASK]` 치환은 문맥을 깨뜨려 답변 품질이 떨어짐

## 기본 아이디어

### 1. 프롬프트 가로채기
- 사용자의 프롬프트를 프록시가 먼저 받는다

### 2. 로컬 민감정보 탐지
- 로컬 NER/PII 탐지기로 사람명, 회사명, 금액, 프로젝트 코드명 등을 찾는다

### 3. 문맥 유지 치환
- 민감정보를 단순 마스킹하지 않고, 의미 구조를 유지하는 대체 값으로 바꾼다
- 예:
  - `삼성전자` → `[Company_A]` 또는 동적 가짜 회사명
  - `10억 원` → `[Amount_1]` 또는 의미 범주가 맞는 대체 금액
  - `강철원` → `[Person_X]`

### 4. 외부 LLM 호출
- 치환된 안전한 프롬프트만 외부 LLM에 전송한다

### 5. 역치환 복원
- 응답을 받은 뒤 세션별 매핑 테이블을 이용해 원래 정보로 복원한다

## 발전된 방향

### 동적 휘발성 매핑
- 고정 패턴 치환은 빈도 분석과 문맥 추론에 취약할 수 있음
- 세션마다 랜덤한 매핑 규칙을 만든다
- 응답 복원 후 매핑 테이블은 즉시 폐기하거나 짧은 TTL 후 삭제한다

### 문맥 유지
- 아무 단어나 랜덤으로 넣는 게 아니라
- 사람명/회사명/금액/프로젝트명처럼 타입을 맞춰 치환해야 한다
- 그래야 외부 LLM의 추론 성능이 크게 무너지지 않는다

## 기술적 난점

- 고정 토큰과 랜덤 가명 사이의 품질/보안 균형
- 원문과 치환문 간 의미 왜곡 방지
- 숫자/단위/날짜/코드명/약어 처리
- 원문에 실제 대체값과 충돌하는 경우
- 코드 블록, 표, 계약서 문장, 회의록 등 구조화된 텍스트 처리
- 역치환 실패 시 잘못된 정보 반환 위험

## MVP 범위

처음부터 완전한 범용 제품을 만들지 않는다.

### MVP 1단계
- 텍스트 프롬프트 전용
- 한국어 업무 문장 위주
- 회사명/인명/금액/프로젝트명 4종만 처리
- 세션별 매핑 테이블 생성/복원
- 외부 LLM 1개와만 연동

### MVP 2단계
- 문서 업로드
- 테이블/숫자 처리 강화
- 감사 로그, 정책 엔진, 관리자 설정 추가

## 제품 포지션

- SaaS/API 게이트웨이
- 엔터프라이즈 온프레미스 보안 프록시
- 사내 LLM 보안 계층

## 왜 파디엠과 맞는가

- AI/SW/데이터 기반 회사 포지션과 맞음
- 엔지니어링 난도가 높아 차별화가 가능함
- 단순 서비스보다 기술형 B2B 제품으로 확장 가능
- 기존의 공공안전/보안 문제의식과도 연결 가능

## 주의

- `완벽한 보안`으로 표현하면 안 됨
- 보안 제품은 기술 데모보다 정책 통제, 감사, 재현성, 오탐/누락 관리가 중요함
- 아이디어는 강하지만, 실제 제품화 전엔 반드시 작은 범위로 기술 검증부터 해야 함
